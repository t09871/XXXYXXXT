CURRENT PROJECT IDENTITY
HBMR / Birdbill is a hummingbird identity reconstruction system. It has evolved from a crop review pipeline into a pose-aware, evidence-ranked re-ID architecture. The current phase is transitioning from isolated crop review toward temporal evidence-aware sampling, where video sequences, adjacent frames, pose continuity, and high-quality still-photo evidence are used to select better identity evidence before human review or model training.

CORE PIPELINE
Current broad architecture: video or still-photo source -> SamplerGUI evidence intake -> frame/photo candidate generation -> MegaDetector crop detection -> MMPose anatomical localization -> AutoRefine evidence crops -> bill/head/gorget/tail/body evidence scoring -> bill base/tip trainer or LightGlue/AutoSort query-vs-reference workflow -> Identity Lab / cleanupGUI reconciliation -> profiles and future ecological/social outputs. The key current shift is that SamplerGUI should become the front-end evidence generator before identity reconciliation.

MODULE MAP
main.py owns legacy/core video intake, settings loading, frame sampling, MegaDetector orchestration, crop_queue insertion, and database/review artifacts. megadetector.py owns MDV6 detector loading, detection normalization, crop padding, and output/crops export. visualid.py owns DINOv2 embedding logic but DINO is currently de-emphasized because it grouped too much by background/lighting. autoname.py owns all provisional Bird##### allocation and no other module should generate Bird IDs. cleanupGUI.py is the current Identity Lab-style reconciliation workbench for provisional identity clusters, crop-level moves/splits, false positives, low-quality, junk, and manual sorting. profiles.py owns derived profile pages and galleries. speciesid.py is technically functional but biologically unreliable for current hummingbird crops and is optional/deprioritized. mmposeGUI.py is the standalone OpenMMLab/MMPose debugging and pose-localization environment. autorefineGUI.py is the pose-derived anatomical evidence crop module. autosortGUI.py is the LightGlue query/reference ranking workflow. billtipTrainerGUI.py is the human-corrected bill base + bill tip labeling tool. SamplerGUI is the new active evidence intake/sampling tool.

DATA / SCHEMA / STORAGE PHILOSOPHY
The current data-era shift is from isolated crop rows toward source-aware sequences. Future rows should preserve source_type, source_path, source_video_key, frame_number, time_sec, sequence_id, burst_id, neighbor_count, pose_stability_score, best_bill_frame, best_gorget_frame, best_tail_frame, identity_evidence_score, and crop-edge/visibility quality fields where applicable. Sampled frames should be treated as cache/intermediate state, not permanent archive by default. Save selectively: preserve best evidence crops and metadata, avoid duplicating originals, use source-path/frame keys or hashes to avoid repeat crops, and separate best evidence from disposable intermediate data. Still-photo imports are important because long-lens photos may provide higher-resolution identity reference evidence than slow-motion video crops.

UI / USER WORKFLOW PRINCIPLES
The user strongly prefers low typing, drag/drop or GUI workflows, selected-file imports rather than giant folder scans, low repetitive labeling, and smooth review tools. The guiding principle is low user workload with high-quality evidence. Training should be made less tedious by using SamplerGUI to find likely-good bill/head/gorget/tail crops before opening labelers. Human review should verify high-value candidates, not sift through large amounts of weak evidence.

VALIDATED LESSONS AND PITFALLS
Evidence quality matters more than raw crop quantity. MMPose does not need perfect hummingbird semantics; it needs stable repeatable localization. Existing AP-10K animal pose priors transfer surprisingly well to hummingbird anatomy. Bill base correction matters because MMPose/AP-10K nose is often not true bill base. DINOv2 is not reliable as a default individual discriminator because it can over-weight background, lighting, and crop composition. LightGlue is useful as a local verifier/ranker, especially in query-vs-reference workflows, but not as a monolithic identity solution. All-vs-all matching can rediscover adjacent frames from the same video instead of true cross-video identity. OpenMMLab dependencies are fragile and should remain isolated in openmmlab-env with explicit interpreter paths. Storage ballooning is a real risk, especially sampled frames and backup-per-action undo behavior.

Deferred: hand posture tracking, direct VHR-BirdPose integration, SLEAP/LEAP experiments, 3D-MuPPET-style full integration, stereo feeder hardware build.

UV/multispectral re-ID, smart feeder systems, field stations, education/exhibit mode, and verified hummingbird identity datasets remain long-term sideproject directions. CURRENT PROJECT IDENTITY AND STRATEGIC PHASE
HBMR / Birdbill is now in a DLC billtip bootstrap phase. The immediate goal is to wrap up the bill-tip detour as quickly as possible and convert it into a usable, mostly hands-off ML evidence source for AutoRefine. The guiding principle going forward is: hands off as much as possible. Human work should not become a bulk labeling requirement. Manual input should be limited to spot-checking, correction, validation, and debugging, not clicking through hundreds or thousands of crops.

This phase represents the first successful transition of the bill-tip branch from custom heuristic/tooling/debug infrastructure into an actual trainable ML workflow. MMPose/AP-10K remains useful as an off-the-shelf pose localization tool, but the new bill-tip path is DeepLabCut-based and independent.

CORE PIPELINE / DATA FLOW
Current intended flow:

video/photo archive input
→ samplerGUI.py sparse evidence intake
→ MegaDetector detector crops
→ billtipTrainerGUI / Birdbill premium JSON annotation source
→ billtipToDLC.py conversion bridge
→ DeepLabCut project training
→ DLC bill_base / bill_tip prediction
→ confidence + geometry gating
→ AutoRefine anatomical crops
→ like-to-like comparison such as gorget-to-gorget, head-to-head, bill-side-to-bill-side
→ AutoSort candidate ranking
→ Identity Lab / cleanupGUI reconciliation

The immediate bill-tip path is:

Birdbill premium JSON
→ DLC labeled-data folder
→ DLC training dataset
→ DLC-trained 2-keypoint model
→ DLC predictions on detector crops
→ Birdbill import/overlay/AutoRefine usage later

Critical strategic rule:
Do not require future humans to bulk-label 500+ images. The current 118 premium rows are enough to train a first DLC model and test whether bill-tip predictions are useful. If the model is imperfect, use high-confidence / geometrically sane predictions and discard weak outputs. Bill-tip should be an opportunistic evidence source, not a blocker.

MODULE MAP
main.py owns legacy/core video intake, review/database pipeline, sampling, MegaDetector orchestration, crop_queue insertion, and visualid assignment in the older identity pipeline.

megadetector.py owns MDV6 detector loading, normalized detections, crop export, and the canonical output/crops crop contract.

samplerGUI.py owns archive intake, sparse frame/photo preparation, detector orchestration, detector-crop manifests, evidence scoring, and bridge routing into billtip training.

billtipTrainerGUI.py owns human-supervised bill-base + bill-tip training data creation and Birdbill JSON export. It remains the Birdbill-side annotation/provenance tool, but should not become a long-term bulk-labeling bottleneck.

billtipToDLC.py is the new single-purpose conversion bridge. It reads Birdbill premium JSON and writes DLC-compatible labeled-data files. It does not train, does not launch DLC GUI, does not touch the Birdbill database, and does not own annotation truth. It is a generated bridge from Birdbill annotation format into DLC training format.

DeepLabCut owns actual bill_base / bill_tip model training and prediction for this branch. DLC is not replacing Birdbill. DLC is the external ML engine.

mmposeGUI.py remains a standalone pose-debugging visualizer and should not be called by other GUIs.

MMPose/AP-10K remains useful for pose localization and approximate anatomical anchors, but the DLC bill-tip model is independent and is not a layer of MMPose.

Future likely module: dlcBilltipInfer.py or billtipInfer.py, a headless DLC inference bridge that runs DLC prediction over crop folders, writes overlays/prediction CSV or JSON, and lets Birdbill consume predictions without using the DLC GUI.

visualid.py owns DINOv2 identity embeddings but remains deprioritized for final identity confidence because DINO had background/lighting bias.

autorefineGUI.py owns anatomical evidence refinement concepts, including future crops for head, throat/gorget, body, tail, and bill-side evidence.

autosortGUI.py owns local candidate ranking / same-bird proposal workflows, likely using LightGlue and future fusion scoring.

cleanupGUI.py remains the Identity Lab reconciliation workspace.

   Bill-tip should become confidence-gated automation, not a perpetual annotation task.

8. Speed answers have been dangerous for this project.
   Future work should prioritize careful checking over fast assumptions.

9. Existing options should be researched before custom work.
   The project lost time by reinventing pieces that may already exist in the ML landscape.
