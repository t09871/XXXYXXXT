HIGH-LEVEL ARCHITECTURE DIRECTION
=================================

Birdbill is transitioning from:
global crop similarity and ad hoc module outputs

toward:
pose-aware localized biological evidence reconstruction.

Current strongest conceptual pipeline:

source video
→ sampled frames
→ MegaDetector bird crops
→ broad pose/keypoint scaffold
→ DLC billtip precision keypoints
→ labeled observation record
→ AutoRefine anatomical evidence crops
→ biometrics / local verification / candidate ranking
→ Identity Lab reconciliation
→ profiles and derived reports

Core principle:
observe first, reconcile later.

The system should support large autonomous ingestion while preserving uncertainty. Human work should be used mainly for correction, validation, reconciliation, and debugging, not as a blocking default workflow.

========================================================================
STORAGE DISCIPLINE
==================

Storage discipline is now a major rebuild driver.

Observed problem:
output\crops ballooned to roughly 24k images and about 10 GB with only limited input processed.

Architectural conclusion:
output\crops currently does too many jobs:
detector evidence, debug output, training data, profile evidence, processing cache, duplicate archive, and possible durable evidence.

Future architecture must separate:

* source media
* sampled-frame cache
* detector crop cache
* durable evidence crops
* refined evidence crops
* training data
* debug outputs
* reports
* derived profile exports

Canonical truth:
source media is the canonical recoverable truth.

Derived/cache state:
sampled frames and temporary crops.

Durable evidence:
selected crops, refined evidence crops, validated profile evidence, BirdImager captures, and biometrics-ready outputs.

Future requirement:
do not keep every generated crop/frame forever by default.

Potential retention policy:
delete sampled frames after crop generation unless frame retention is enabled for debugging.

DLC BILLTIP CONTRACT
====================

DLC billtip is now validated enough to treat as a real Birdbill evidence source.

→ MegaDetector bird crop
→ DLC bill_base / bill_tip
→ bill vector
→ bill length in pixels

Current DLC keypoint meanings:

bill_base
bill_tip

DLC owns:
precise bill_base and bill_tip localization on usable bird crops.

MegaDetector owns:
whether the crop exists and whether the crop contains a usable bird.

Current rule:
freeze the validated two-point DLC billtip model as a specialist asset.

Do not casually expand this model in-place.

If body/wing landmarks are pursued later:
define a careful landmark contract first and likely create a separate DLC project such as birdpose or bodywing.

billtip is now a known-good specialist branch and should not be destabilized by broader ambiguous labeling.

DLC and MMPose are complementary, not competitors.

MMPose:
broad scaffold and anatomical localization.

DLC billtip:
precision specialist for bill_base, bill_tip, bill vector, bill-side crop, and bill-length evidence.

Future integrated flow:

BIRDIMAGER / BIRD3D CONTRACT
============================

Current status:
standby / architectural placeholder.

Do not implement now.

Do not write BirdImager-specific code yet.

Do not build a separate BirdImager landmark-training path.

BirdImager/Bird3D likely belongs inside Birdbill as a module, subprogram, or tightly integrated workflow rather than a separate pipeline.

Core idea:
dual-camera evidence capture using two independent cameras aimed at the same feeder/perch area, roughly 45–70 degrees apart, with rough target around 60 degrees.

Initial sync:
audio slate spike or other visible/audible sync cue, not hardware triggering.

Architectural rule:
dual-camera footage should be treated both as ordinary Birdbill input and specialty paired evidence.

Each camera stream should run independently through:

raw video
→ autosampler frames
→ MegaDetector crops
→ shared MMPose/DLC keypoints
→ labeled observations

Then BirdImager/Bird3D layer may add:

* shared session ID
* camera/source metadata
* sync offset
* synced timestamps
* crop/keypoint pairing
* calibration metadata
* 3D keypoint triangulation
* biometrics-ready outputs
* optional fitted mesh/STL records

Critical future field:
synced_time_ms or equivalent.

Primary BirdImager/Bird3D value:
triangulated 3D keypoints and identity-relevant biometrics, not STL generation.

Potential measurements:

* bill length
* bill curvature
* bill width/depth estimates
* bill-to-head angle
* head width
* head length
* eye-to-bill-base distance
* crown/throat geometry
* head/bill ratios
* relative orientations
* cross-view consistency

STL/mesh:
experimental downstream possibility only.

Most realistic mesh target:
bill-only STL.

Any mesh must be labeled:
fitted/reconstructed evidence, not a literal perfect scan.

MEGADETECTOR OUTPUT / GEOMETRY CONTRACT
=======================================

MegaDetector output is raw detector evidence, not direct bird truth.

The detector wrapper must preserve all detections and geometry, then classify downstream role.

Suggested downstream roles:

animal:
candidate bird evidence

person:
human/context geometry

vehicle:
likely false positive or irrelevant context

other:
context/review

Do not send person/vehicle detections to DLC billtip.

Do preserve person/vehicle bbox records for:

* human-contact behavior context
* false-positive training
* relationship/proximity geometry
* future hand/finger/perch analysis
* debugging detector behavior

Known behavior:
When a bird is landed on a human finger/hand, full frames may detect person/human and may miss or de-emphasize the bird. Vehicle/other false positives may also appear. This should be handled by Birdbill filtering/retention logic, not treated as automatic detector failure.

Minimum expanded detector geometry/provenance fields:

* source_video
* frame_path
* source_frame_index
* frame_time_seconds
* timestamp / synced timestamp later
* detection_id
* class_name
* class_id
* confidence
* bbox_x1
* bbox_y1
* bbox_x2
* bbox_y2
* bbox_width
* bbox_height
* bbox_center_x
* bbox_center_y
* bbox_area_px
* crop_path if exported
* crop_padding_px
* crop_to_frame_offset_x
* crop_to_frame_offset_y
* crop_to_frame_scale_x
* crop_to_frame_scale_y

Reason:
Future Birdbill relationship and geometry work needs preserved bbox data for:

* bird-to-bird distance
* bird-to-feeder distance
* bird-to-hand/finger bbox distance
* bird-to-perch/bath/flower zone distance
* multibird spacing
* motion vectors
* event-level proximity timing
* future 3D / paired-camera evidence
