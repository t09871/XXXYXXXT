1. CURRENT 10,000-FT BIRDBILL PIPELINE
========================================================================

Birdbill is transitioning from ad hoc crop similarity toward pose-aware localized biological evidence reconstruction.

CURRENT CONCEPTUAL PIPELINE:

source media / raw video
→ Smart Frame Sampler
→ MegaDetector / detector evidence
→ crop generation and crop retention scoring
→ MMPose broad pose / body scaffold / bill-base candidate
→ DLC billtip specialist evidence
→ merged labeled observation record
→ Smart Cropper / AutoRefine anatomical evidence crops
→ biometrics eligibility and measurement signals
→ AutoSort / fusion scoring / candidate ranking
→ Identity Lab / reconciliation GUI
→ profiles, community, reports
→ optional BirdImager/Bird3D paired-camera evidence layer later

Core principle:
observe first, reconcile later.

Human work should be used mainly for correction, validation, reconciliation, and debugging, not as a blocking default workflow.

IMPORTANT PIPELINE POSITION CORRECTION:
The recent DLC smoke/inspector work was a capability proof, not the real production pipeline position. DLC belongs after crop generation and often after MMPose/body/bill-base context. DLC-only mode may be supported as an optional branch, but the intended strongest full pipeline is MMPose scaffold → DLC billtip → merged labeled observation → AutoRefine.

OPTIONAL POSE MODES BIRDBILL SHOULD EVENTUALLY SUPPORT:

1. MMPose-only mode
   Role: broad body/keypoint hints; weak for exact hummingbird bill parts but useful as anatomical scaffold.

2. DLC-only mode
   Role: specialist keypoints when a DLC project is trained for the relevant crop/frame style. Can run on MegaDetector crop outputs.

3. MMPose → DLC billtip mode
   Role: MMPose supplies broad body/bill-base scaffold; DLC refines bill_tip and/or precise bill_base. Likely strongest for bill measurement but should not be the only possible path.

4. Future paired-camera / 3D mode
   Role: paired 2D keypoints, sync metadata, triangulation. Not minimum functional Birdbill.

ARCHITECTURE RULE:
Detection produces bird candidates. Pose modules attach evidence to candidates. AutoRefine consumes whichever evidence exists. Fusion scoring consumes refined crops plus evidence. GUI allows inspection, correction, and commit.

Do not hardcode production Birdbill as: detection → MMPose → DLC → everything else must depend on this exact chain. That path is too brittle.

output\crops ballooned to roughly 24k images and about 10 GB with only limited input processed.

Architectural conclusion:
Old output\crops did too many jobs: detector evidence, debug output, training data, profile evidence, processing cache, duplicate archive, and possible durable evidence.

source media
sampled-frame cache
detector crop cache
durable evidence crops
refined evidence crops
training data
debug outputs
reports
derived profile exports

Canonical truth:
source media is canonical recoverable truth.

Derived/cache state:
sampled frames and temporary crops.

Durable evidence:
selected crops, refined evidence crops, validated profile evidence, BirdImager captures, biometrics-ready outputs.

Future requirement:
do not keep every generated crop/frame forever by default.

Potential retention policy:
delete sampled frames after crop generation unless frame retention is enabled for debugging.

SmartFrameSampler v0.3 explicitly marks preview frames and frame records purgeable and non-durable.
